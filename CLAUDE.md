## Project Overview

**mobiscroll-demos** is a meta-repository containing demo applications for the Mobiscroll UI components library across 7 different JavaScript frameworks:
- Angular (v21)
- JavaScript (vanilla)
- jQuery
- React (v19)
- React + TypeScript
- Vue 3
- Vue 3 + TypeScript

Each framework has its own subdirectory (`mobiscroll-demos-{framework}`) with a complete, independent demo application. The repository uses utility scripts to manage demos across all frameworks simultaneously.

## Repository Structure

### Root Level
- **package.json**: Contains root-level lint script and dependencies for utility scripts (fs-extra, mssql)
- **config.json**: Configuration pointing to external public demo repositories
- **Utility scripts** (Node.js):
  - `generate.js`: Queries Azure SQL database to generate demo lists and metadata
  - `add.js`: Scaffolds new demo files across all 7 frameworks using templates
  - `rename.js`: Renames/relocates demos while maintaining consistent structure across frameworks
  - `copy.js`: Syncs demos to separate public repositories
  - `readme.js`: Auto-generates README.md files for individual demos with links

### Demo Organization
Each framework's `src/demos/` directory follows this structure:
```
demos/
├── datepicker/
│   ├── calendar/     (various demo folders)
│   ├── datetime/     (various demo folders)
│   └── range/        (various demo folders)
├── eventcalendar/    (various demo folders)
├── pickers/          (various demo folders)
├── form-components/  (various demo folders)
└── Demos.js          (or equivalent) - Auto-generated registry importing all demos
```

Each individual demo lives in its own directory: `category/component/demo-name/` containing:
- Framework-specific implementation file (`.jsx`, `.tsx`, `.vue`, `.js`, `.ts`)
- `README.md` with description and links to live demo
- Optional CSS files

### Shared Code
All frameworks include:
- `src/addons/calendar-integration/` - File-based dependency addon
- `src/addons/print/` - File-based dependency addon
- `src/pages/` - Page/routing components
- Main entry point (App.jsx, App.vue, etc.)

## Building & Running

### All Frameworks - Linting
Lint all frameworks from the root:
```bash
npm run lint
```

This runs ESLint across all 7 frameworks with framework-specific rules.

### Vite-Based Frameworks (React, React-TS, Vue, Vue-TS, JavaScript, jQuery)
Build tool: **Vite v6**

From the framework subdirectory:
```bash
npm run dev       # Start dev server (http://localhost:5173)
npm run build     # Production build to dist/
npm run lint      # Framework-specific ESLint
npm run preview   # Preview production build locally
```

### Angular Framework
Build tool: **Angular CLI v21**

From the `mobiscroll-demos-angular` subdirectory:
```bash
npm start                    # ng serve (dev server)
npm run build                # ng build (production)
npm run watch                # ng build --watch --configuration development
npm run test                 # ng test (Karma + Jasmine)
npm run lint                 # Angular ESLint
npm run serve:ssr:mobiscroll-demos-angular  # SSR server
```

## Development Workflow

### To Work on a Specific Framework
1. Navigate to the framework directory: `cd mobiscroll-demos-react` (or relevant framework)
2. Ensure dependencies are installed: `npm install`
3. Start dev server: `npm run dev`
4. Edit demo files in `src/demos/` or main app components in `src/`

### To Add a New Demo
Use the `add.js` script from the root directory:
```bash
node add.js --component datepicker --category calendar --name my-new-demo
```
This scaffolds the demo across all 7 frameworks with proper templates.

### To Rename/Move an Existing Demo
From root:
```bash
node rename.js "old-category/old-component/old-demo-name" "new-category/new-component/new-demo-name"
```
This updates the demo path consistently across all frameworks.

### To Sync Demos to Public Repositories
From root:
```bash
node copy.js
```
This copies all framework demo directories to the external public repositories defined in `config.json` and commits/pushes changes.

## Key Architectural Insights

### Demo Registry System
Each framework has a **1000+ line auto-generated `Demos.js` file** that:
- Imports all demo components from the nested directory structure
- Exports a hierarchical data structure organizing demos by category, component, and specific demo
- Used by the routing/navigation system to dynamically render available demos
- Typically regenerated by the `generate.js` script from database metadata

Example structure from `Demos.js`:
```javascript
{
  name: 'Datepicker',
  items: [
    {
      name: 'Calendar',
      items: [
        { name: 'Formatting return values', unique: 'formatting-return-values', component: Demo332 },
        // ... many more
      ]
    }
  ]
}
```

### Multi-Framework Parity
Demos are intentionally kept consistent across frameworks. The same demo exists in all 7 variants but implemented in the framework-specific way:
- React: JSX components using Mobiscroll React library
- Vue: Vue SFC templates using Mobiscroll Vue library
- Angular: Angular components with TypeScript
- JavaScript: Vanilla JS with Mobiscroll JavaScript library
- jQuery: jQuery integration with Mobiscroll jQuery library
- React-TS & Vue-TS: TypeScript versions of React/Vue

### Database-Driven Demo Generation
The `generate.js` script connects to an Azure SQL database to:
- Fetch demo metadata and categorization
- Map database entries to component groups (datepicker, eventcalendar, etc.)
- Generate structured demo lists

This means demo changes can originate from a CMS/database rather than purely code-based.

### Shared Addons Pattern
Each framework includes `src/addons/` directory with file-based local dependencies:
- `package.json` specifies: `"@mobiscroll/calendar-integration": "file:src/addons/calendar-integration"`
- These are local npm packages within the project
- Shared across all framework implementations

## Development Notes

- **No GitHub Actions/CI workflows** are configured (no `.github/workflows/` directory)
- **Database credentials** are hardcoded in `generate.js` (Azure SQL connection) - this is for internal development only
- **Demo README generation** is automated via `readme.js` - individual demo READMEs should not be manually edited (they'll be overwritten)
- **Git remotes** point to both the main repo and separate public repos for each framework
- **TypeScript frameworks** (react-ts, vue-ts) require TypeScript compilation before build: `npm run build` includes `tsc` step
